# рЯ§Ц New Business Onboarding Protocol

**Audience:** AI Agent (Claude or successor) supporting AI Accounting Template users
**Purpose:** End-to-end workflow from "I need to set up a new business" вЖТ fully rendered workspace ready for first JE
**Status:** Phase 6 deliverable. Reference document for AI agents вАФ follow verbatim when user requests onboarding.

---

## рЯОѓ When this protocol applies

Trigger if user says any of the following (or equivalent in Vietnamese):

- "Set up a new business / company / client"
- "I just started an LLC / S-Corp / etc. вАФ help me onboard"
- "Create a new instance in this template"
- "Onboard [Company Name]"
- "I'm a fractional CFO вАФ add a new client"

**Do NOT** trigger for: "add a bank account", "add an employee", "update our info" вАФ those are single-field updates to an existing `_BUSINESS_CONFIG.json`, not full onboarding.

## рЯУЛ The 7-step protocol

### Step 1 вАФ Confirm intent + timeline

Before launching wizard, confirm with user:

> "I can help you onboard [company/new business] into this template. The setup takes about 5 minutes вАФ 15 questions about entity type, state, fiscal year, owner, and compliance status. At the end you'll have:
> 1. A `_BUSINESS_CONFIG.json` (single source of truth for the business)
> 2. An `ONBOARDING_CHECKLIST.md` (30 items tailored to your entity/state/industry)
> 3. A fully rendered workspace under `_INSTANCES/{instance_id}/`
>
> Ready to start?"

If user confirms вЖТ Step 2. If they want to skip the wizard (e.g., they already have the JSON), go to Step 4.

---

### Step 2 вАФ Direct user to wizard

Tell the user:

> "Open `_CONFIG/setup_wizard.html` in your browser (double-click it, or drag to Chrome/Edge). Work through the 6 steps. When done, click **Download JSON** and **Download Checklist** вАФ I'll use those files to render your workspace."

**Do NOT** attempt to fill the wizard on the user's behalf вАФ it's a browser form, not a file AI can edit directly. The wizard handles validation, preset shortcuts, and file download.

**If user can't open HTML** (e.g., claude.ai web interface without filesystem): offer to collect the same 15 questions conversationally, then output JSON directly. Use the schema at `_CONFIG/_BUSINESS_CONFIG.schema.json` as the spec.

---

### Step 3 вАФ Wait for JSON

User will paste the downloaded JSON or attach it. Parse and validate:

1. **Schema validation** вАФ Must match `_CONFIG/_BUSINESS_CONFIG.schema.json`:
   - `schema_version === "1.0.0"`
   - All required fields present (see schema `required` arrays)
   - Enum values valid (entity_type, state_of_operation, industry, etc.)
   - Regex patterns valid (EIN `XX-XXXXXXX`, ZIP, instance_id `^[A-Za-z0-9_-]{3,50}$`)

2. **Semantic validation** (things schema can't catch):
   - `fiscal_year.end_date` is after `fiscal_year.start_date`
   - `tier_1_max < tier_2_max < tier_3_max`
   - `state_of_incorporation` + `state_of_operation` make sense together
   - If `is_owner_employee = true` and `entity_type = S-Corp` вЖТ flag reasonable-comp setup as critical

If validation fails вЖТ tell user exactly which field, what's wrong, what valid values look like. Do **NOT** proceed to rendering with invalid config.

### Step 5 вАФ Render MD templates into instance

Apply the 7-pass render algorithm from `_CORE/RENDER_PROTOCOL.md` against these 5 core templates:

| Template (source) | Rendered output |
|-------------------|-----------------|
| `_CORE/README.md.template` | `_INSTANCES/{instance_id}/README.md` |
| `_CORE/AI_AGENT_MEMORY.md.template` | `_INSTANCES/{instance_id}/AI_AGENT_MEMORY.md` |
| `_CORE/COMPANY_PROFILE.md.template` | `_INSTANCES/{instance_id}/COMPANY_PROFILE.md` |
| `_CORE/FORM_SCHEMA.md.template` | `_INSTANCES/{instance_id}/FORM_SCHEMA.md` |
| `_CORE/USE_CASES_GUIDE.md.template` | `_INSTANCES/{instance_id}/USE_CASES_GUIDE.md` |

**Before rendering, verify dependent modules exist:**
- `_MODULES/entity/{entity_type}.md` вАФ required. All 6 entity modules shipped.
- `_MODULES/state/{state}.md` вАФ required for detailed states (CA, NY, TX, FL, DE, WA, IL, GA, NJ, MA). For other states, use **Lenient mode** from RENDER_PROTOCOL.md: render INCLUDE fallbacks with a warning comment, don't block.

**Render checklist:**
1. Resolve all `{{FIELD}}` placeholders from config
2. Resolve all `{{FORMAT:...|formatter}}` directives
3. Resolve all `{{COMPUTED:...}}` values (TODAY, CURRENT_QUARTER, FY_PROGRESS_PCT, PRIOR_FY, etc.)
4. Resolve all `{{INCLUDE:entity/{entity_type}.md#section-id}}` directives
5. Resolve all `{{INCLUDE:state/{state}.md#section-id}}` directives
6. Evaluate all `{{IF:...}}{{ENDIF}}` blocks
7. Grep output for `{{` вАФ **must return 0 matches** before calling render complete

## рЯЫС Do NOT

- **Do NOT** assume defaults for entity_type, state, or fiscal_year вАФ these are high-consequence choices. Always get explicit user input.
- **Do NOT** render if validation fails. Invalid config вЖТ broken downstream files.
- **Do NOT** copy another instance's data (customer names, JEs, etc.) into the new instance. Each instance starts empty.
- **Do NOT** modify `_CORE/*.md.template` or `_MODULES/*` files during onboarding вАФ those are shared templates. Only write to `_INSTANCES/{instance_id}/`.
- **Do NOT** skip the checklist. Even experienced business owners miss BOI filing or state registration renewal. The checklist is the handoff artifact.
- **Do NOT** substitute your own knowledge for what's in `_MODULES/state/{state}.md`. Tax rates change annually. Use the state module as source of truth вАФ if it's stale, update the module, don't freehand the render.

## вЪ†пЄП Edge cases

### User wants to onboard multiple clients (fractional CFO use case)
Run the protocol once per client. Each gets its own `_INSTANCES/{client_id}/`. The HTML forms support a config switcher (вЪЩпЄП button) to flip between loaded clients.

### State is not in our 10 detailed states
Proceed in **Lenient mode**: render everything that doesn't depend on the state module, and leave clearly-marked TODOs (e.g., `<!-- TODO: State module for OR not built. Confirm OR franchise/income tax rules manually. -->`) where state-specific content would go. Flag this to user in Step 7.

### Custom fiscal year (non-calendar)
Wizard supports this. Make sure FY start/end flow through to `Journal_Entries/FY{year}_Q{1-4}/` folder naming вАФ quarters are based on FY, not calendar.

### Missing required config field
Wizard enforces required fields but if user provides JSON manually (skipping wizard), validate strictly. If `company.legal_name` missing вЖТ refuse to render, ask for it.

### User wants to re-render after editing config
Safe. The render is idempotent вАФ re-running against an updated config overwrites the 5 MD files + checklist but leaves data folders (00вАУ07) and their contents untouched. Always confirm before re-rendering: *"This will overwrite README.md, AI_AGENT_MEMORY.md, COMPANY_PROFILE.md, FORM_SCHEMA.md, USE_CASES_GUIDE.md, ONBOARDING_CHECKLIST.md. Your data folders (JEs, invoices, receipts) are untouched. Proceed?"*

### User wants to migrate an existing demo-like workspace into the template structure
This is Phase 7 territory вАФ different protocol. Direct user to Phase 7 docs when ready.

## рЯУЮ Handoff

After Step 7, the user has a functioning workspace. Subsequent AI agent invocations for the same business should:

1. Read `_INSTANCES/{instance_id}/_BUSINESS_CONFIG.json` first (single source of truth)
2. Read `_INSTANCES/{instance_id}/AI_AGENT_MEMORY.md` second (quick context)
3. Then respond to whatever the user is actually asking
